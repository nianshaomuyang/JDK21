零、为什么是 JDK 21？

很多团队还停留在 JDK 8。升级的动力是什么？升到哪个版本？成本多大？

第一步：为什么要升？→ Spring Boot 3 + GraalVM

Spring Boot 3 支持 GraalVM Native Image，可以将 Java 应用编译成原生二进制：

指标传统 JVMGraalVM Native Image差距
启动时间1-3 秒10-50ms50-100x
内存占用200-300MB30-50MB5-6x
首次请求延迟1-2 秒30-50ms30-50x
镜像大小300MB+30-50MB6-10x
✅ 对 Serverless、微服务弹性伸缩、容器化部署场景，这是质变：
• AWS Lambda 冷启动从 3 秒降到 30 毫秒
• K8s Pod 扩缩容速度提升 50 倍
• 容器镜像从 300MB 缩到 50MB，拉取速度提升 6 倍

第二步：Spring Boot 3 要求 JDK 17+

Spring Boot 3.0 最低要求 JDK 17，Spring Boot 3.2 也只需要 JDK 17。

所以升级路径是：JDK 8 → JDK 17

第三步：那升到 JDK 17 就够了？

不够。JDK 21 有 4 个独占特性，JDK 17 没有：

Record Patterns解构赋值，替代 Visitor 模式AST 处理、数据解析
Sequenced Collections统一的首尾访问 API日常集合操作

虚拟线程的价值尤其大——它让并发编程回归简单：

1.1 Record：不可变数据载体

JDK 14 Preview JDK 16 Final
📌 演进：JDK 14 引入 Preview → JDK 16 正式定稿 → JDK 21 中作为 Pattern Matching 的基础

写法与编译后对比

// 一行定义不可变数据类
public record User(String name, int age, String email) {}

用 javap -p User.class 反编译，看看编译器生成了什么：

public final class User extends java.lang.Record {
    // 所有字段都是 private final（不可变！）
    private final String name;
    private final int age;
    private final String email;

    // 全参构造器
    public User(String name, int age, String email) {
        this.name = name;
        this.age = age;
        this.email = email;
    }

    // 访问器（不是 getXxx，直接用字段名）
    public String name() { return name; }
    public int age() { return age; }
    public String email() { return email; }

    // 自动生成 equals() / hashCode() / toString()
}

final 语义：为什么 Record 是不可变的

record User(String name, int age) {}

User user = new User("张三", 28);

// ❌ 编译报错：final 字段不能赋值
// user.name = "李四";

// ❌ 没有 setter 方法
// user.setName("李四");

// ✅ 想修改？返回新对象（函数式风格）
record User(String name, int age) {
    User withName(String newName) {
        return new User(newName, this.age);  // 返回新对象，原对象不变
    }
}

User user2 = user.withName("李四");
// user 不变，user2 是新对象
✅ final 的价值： 线程安全（不可变对象天然线程安全）· 哈希安全（可安全用作 Map key）· 防御性编程（不会被意外修改）

compact 构造器：参数校验

record Order(String orderId, BigDecimal amount) {
    // compact 构造器：参数校验
    Order {
        if (orderId == null || orderId.isBlank())
            throw new IllegalArgumentException("订单号不能为空");
        if (amount == null || amount.compareTo(BigDecimal.ZERO) <= 0)
            throw new IllegalArgumentException("金额必须大于 0");
        // 校验通过后，编译器自动执行 this.orderId = orderId 等赋值
    }
}

1.2 switch + 类型模式匹配 + when

JDK 14 箭头语法 JDK 17 Preview JDK 21 Final
📌 演进：JDK 7 只支持 int/String/enum → JDK 14 箭头语法 → JDK 17 模式匹配 Preview → JDK 21 + when 守卫条件 Final

核心语法 ①：箭头语法（JDK 14）

不需要 break，不会穿透，switch 可以作为表达式返回值。

Before — 传统 case/break
String result;
switch (day) {
    case "Mon":
    case "Tue":
    case "Wed":
    case "Thu":
    case "Fri":
        result = "工作日";
        break;          // 必须 break，否则穿透
    case "Sat":
    case "Sun":
        result = "周末";
        break;
    default:
        result = "未知";
        break;
}
After — 箭头语法
// 不需要 break，不会穿透
// switch 作为表达式返回值
String result = switch (day) {
    case "Mon", "Tue",
         "Wed", "Thu",
         "Fri" -> "工作日";
    case "Sat", "Sun" -> "周末";
    default -> "未知";
};

核心语法 ②：类型模式匹配（JDK 17 Preview → JDK 21 Final）

switch 可以判断类型，并自动绑定变量。替代冗长的 instanceof 链。

Before — instanceof 链
if (obj instanceof String s) {
    return "字符串: " + s
        + " (长度" + s.length() + ")";
} else if (obj instanceof Integer i) {
    return "整数: " + i;
} else if (obj instanceof Double d) {
    return "浮点数: " + d;
} else if (obj instanceof Boolean b) {
    return "布尔: " + b;
} else {
    return "其他";
}
After — 类型模式匹配
return switch (obj) {
    case String s ->
        "字符串: " + s
            + " (长度" + s.length() + ")";
    case Integer i -> "整数: " + i;
    case Double d  -> "浮点数: " + d;
    case Boolean b -> "布尔: " + b;
    case null      -> "null";
    default        -> "其他";
};

核心语法 ③：when 守卫条件（JDK 21 Final）

在类型匹配后进一步过滤。注意：when 条件不参与穷举检查，更具体的放前面。

Before — if-else 嵌套
if (num instanceof Integer i) {
    if (i > 0) {
        return "正整数: " + i;
    } else if (i < 0) {
        return "负整数: " + i;
    } else {
        return "零";
    }
} else if (num instanceof Double d) {
    if (d > 0) {
        return "正浮点: " + d;
    } else if (d < 0) {
        return "负浮点: " + d;
    } else {
        return "零点零";
    }
} else {
    return "其他数字";
}
After — when 守卫条件
return switch (num) {
    case Integer i
      when i > 0 -> "正整数: " + i;
    case Integer i
      when i < 0 -> "负整数: " + i;
    case Integer _ -> "零";
    case Double d
      when d > 0 -> "正浮点: " + d;
    case Double d
      when d < 0 -> "负浮点: " + d;
    case Double _ -> "零点零";
    default -> "其他数字";
};
⚠️ 注意事项： 模式顺序很重要 · when 条件不参与穷举检查 · switch 表达式必须穷举 · 箭头语法不需要 break · case null 可以显式处理 null

1.3 sealed class：编译器穷举检查

JDK 15 Preview JDK 17 Final
📌 演进：JDK 15 Preview → JDK 17 正式定稿

核心语法

// 限定子类型：只允许这 4 种支付结果
public sealed interface PayResult
    permits PaySuccess, PayFailed, PayPending, PayRefunded {}

public record PaySuccess(String orderId, BigDecimal amount) implements PayResult {}
public record PayFailed(String reason, boolean retryable) implements PayResult {}
public record PayPending(int estimatedSeconds) implements PayResult {}
public record PayRefunded(BigDecimal refundAmount) implements PayResult {}

// switch 覆盖所有 permits → 不需要 default
return switch (result) {
    case PaySuccess s   -> "成功: " + s.orderId();
    case PayFailed f    -> "失败: " + f.reason();
    case PayPending p   -> "处理中: " + p.seconds() + "s";
    case PayRefunded r  -> "已退款: " + r.amount();
};
// 如果新增 PayTimeout？所有未处理的 switch 编译报错！

子类的三种形式

形式含义使用场景
final不能再被继承最常用，配合 Record
sealed子类也是 sealed继续限定子类型
non-sealed放开限制不推荐，破坏穷举性

1.4 Record Patterns：解构赋值

JDK 19 Preview JDK 21 Final
📌 演进：JDK 19 Preview → JDK 21 正式定稿（JEP 440）

核心语法

record Point(int x, int y) {}
record Line(Point start, Point end) {}
record Circle(Point center, double radius) {}

Before / After 对比

Before — getter 链
// 手动 getter，层层访问
Point s = line.start();
Point e = line.end();
int dx = e.x() - s.x();
int dy = e.y() - s.y();
return Math.sqrt(dx*dx + dy*dy);
After — 解构赋值
// 直接解构，绑定变量
Line(
  Point(var x1, var y1),
  Point(var x2, var y2)
) = line;
int dx = x2 - x1;
int dy = y2 - y1;
return Math.sqrt(dx*dx + dy*dy);

switch 中解构多种类型

return switch (shape) {
    // 单层解构
    case Point(var x, var y) ->
        "(%d, %d)".formatted(x, y);
    // 双层嵌套解构
    case Line(Point(var x1, var y1), Point(var x2, var y2)) ->
        "Line(%d,%d→%d,%d)".formatted(x1, y1, x2, y2);
    // 单层解构 + 基本类型
    case Circle(Point(var cx, var cy), var r) ->
        "Circle(center=(%d,%d), r=%.1f)".formatted(cx, cy, r);
    default -> "unknown";
};

守卫条件 + 解构

return switch (shape) {
    case Circle(Point(var cx, var cy), var r) -> {
        double dist = Math.sqrt(
            Math.pow(p.x()-cx, 2) + Math.pow(p.y()-cy, 2)
        );
        if (dist < r) yield "点在圆内";
        else if (dist == r) yield "点在圆上";
        else yield "点在圆外";
    }
    case Rect(Point(var x1,var y1), Point(var x2,var y2)) -> {
        if (p.x()>=x1 && p.x()<=x2 && p.y()>=y1 && p.y()<=y2)
            yield "点在矩形内";
        else yield "点在矩形外";
    }
    default -> "不支持的图形";
};

实战：工作流审批流程（解构 + 递归）

场景：请假审批流程，根据金额决定是否需要总监审批。

// 定义工作流节点类型
sealed interface WorkflowNode
    permits Start, Submit, Approve, Decision, End {}
record Start(String id, WorkflowNode next)
    implements WorkflowNode {}
record Submit(String id, String applicant,
    double amount, WorkflowNode next)
    implements WorkflowNode {}
record Approve(String id, String approver,
    WorkflowNode next)
    implements WorkflowNode {}
record Decision(String id, String condition,
    WorkflowNode trueBranch, WorkflowNode falseBranch)
    implements WorkflowNode {}
record End(String id, String status)
    implements WorkflowNode {}

// 模拟执行工作流（解构 + 递归 + 条件判断）
static String execute(WorkflowNode node, double amount) {
    return switch (node) {
        case Start(_, var next) ->
            "→ 开始\n" + execute(next, amount);
        case Submit(_, var who, var amt, var next) ->
            "→ " + who + " 提交，金额: " + amt + "\n"
                + execute(next, amt);
        case Approve(_, var approver, var next) ->
            "→ " + approver + " 审批通过\n"
                + execute(next, amount);
        case Decision(_, _, var t, var f) ->
            amount > 5000
                ? "→ 需要总监审批\n" + execute(t, amount)
                : "→ 主管审批即可\n" + execute(f, amount);
        case End(_, var status) ->
            "→ 结束: " + status;
    };
}

// 构建工作流：开始 → 提交 → 主管审批 → (金额>5000 ? 总监审批 : 跳过) → 结束
var workflow = new Start("start",
    new Submit("submit", "张三", 8000,
        new Approve("mgr", "李经理",
            new Decision("check", "金额>5000",
                new Approve("dir", "王总监",
                    new End("end", "审批通过")),
                new End("end", "自动通过")))));

System.out.println(execute(workflow, 8000));
// → 开始
// → 张三 提交，金额: 8000.0
// → 李经理 审批通过
// → 需要总监审批
// → 王总监 审批通过
// → 结束: 审批通过
⚠️ 注意事项： 解构顺序与构造器参数一致 · 嵌套太深影响可读性 · null 会抛 NPE · 性能等价于 getter 调用（编译器语法糖）

1.5 三者组合：替代 Visitor 模式

Before — Visitor 模式（6+ 个类）
interface ExprVisitor<R> {
    R visitLit(LitExpr e);
    R visitAdd(AddExpr e);
    R visitMul(MulExpr e);
}
// 每个子类都要实现 accept()
// 每个操作都要写一个 Visitor
// 6+ 个类/接口
After — sealed + record + switch（几行）
sealed interface Expr
  permits Lit, Add, Mul {}
record Lit(double v) implements Expr {}
record Add(Expr l, Expr r) implements Expr {}
record Mul(Expr l, Expr r) implements Expr {}

static double eval(Expr e) {
    return switch (e) {
        case Lit(var v) -> v;
        case Add(var l,var r) -> eval(l)+eval(r);
        case Mul(var l,var r) -> eval(l)*eval(r);
    };
}
💡 核心价值：sealed + record + switch 三者组合，编译器强制穷举检查，新增类型时所有未处理的 switch 编译报错。这是 JDK 21 类型系统进化的终极体现。

需求：定义支付结果类型，用模式匹配处理不同结果。

Java 21
// sealed + record + switch
sealed interface PayResult
    permits PaySuccess, PayFailed {}
record PaySuccess(String orderId) implements PayResult {}
record PayFailed(String reason) implements PayResult {}

static String describe(PayResult r) {
    return switch (r) {
        case PaySuccess s -> "成功: " + s.orderId();
        case PayFailed f  -> "失败: " + f.reason();

Rust
// enum + match（Rust 最擅长的场景）
enum PayResult {
    Success { order_id: String },
    Failed { reason: String },
}

fn describe(r: &PayResult) -> String {
    match r {
        PayResult::Success { order_id } =>
            format!("成功: {}", order_id),
        PayResult::Failed { reason } =>
            format!("失败: {}", reason),
    }
}
💡 对比总结：
• Rust 的 enum + match 是最强大的模式匹配系统，原生支持穷举检查、解构、守卫条件，语法最简洁
• Java 21 的 sealed + record + switch 是 Java 的重大进步，但语法仍比 Rust 冗长
• Java 的优势在于生态（Spring、海量库）和团队熟悉度，Rust 的优势在于类型系统和性能

2.1 传统模型的痛点

// 这段代码你能跑，但你不想维护它
CompletableFuture.supplyAsync(() -> queryUser(id), pool)
    .thenCompose(user -> CompletableFuture.allOf(
        CompletableFuture.supplyAsync(() -> queryOrders(user.getId()), pool)

2.2 核心机制（源码级）

虚拟线程 vs 平台线程

属性平台线程虚拟线程
栈存储OS 线程栈（~1MB）Continuation（堆内存，按需增长）
创建成本系统调用 + 大块内存Java 对象创建（~几百字节）

Mount / Unmount 机制

// 当虚拟线程遇到阻塞时：
// 1. yield(): 把栈帧「冻结」到堆内存（Continuation）
// 2. unmount(): 从载体线程卸载，载体线程空闲
// 3. 载体线程去执行其他虚拟线程
// 4. 阻塞结束后 remount 继续执行

VirtualThread.run() {
    mount();        // 挂载到载体线程
    task.run();     // 执行用户代码
    unmount();      // 卸载
}

// Continuation: 栈帧冻结与恢复
yield() → 冻结栈帧到堆 → 载体线程释放
run()   → 恢复栈帧到栈 → 继续执行

// 场景：订单详情页并发调用 3 个远程服务（各 200ms）

// Before: CompletableFuture
ExecutorService pool = Executors.newFixedThreadPool(200);
CompletableFuture<OrderVO> future = CompletableFuture
    .supplyAsync(() -> orderService.query(orderId), pool)
    .thenCompose(order -> CompletableFuture.allOf(
        supplyAsync(() -> userService.query(order.getUserId()), pool)
            .thenApply(user -> order.setUser(user)),
        supplyAsync(() -> inventoryService.query(order.getSkuId()), pool)
            .thenApply(inv -> order.setInventory(inv)),
        supplyAsync(() -> couponService.query(order.getCouponId()), pool)
            .thenApply(coupon -> order.setCoupon(coupon))
    ).thenApply(v -> order.toVO()));

// After: 虚拟线程
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    Future<User> userF = executor.submit(() -> userService.query(order.getUserId()));
    Future<Inventory> invF = executor.submit(() -> inventoryService.query(order.getSkuId()));
    Future<Coupon> couponF = executor.submit(() -> couponService.query(order.getCouponId()));

    OrderVO vo = order.toVO();
    vo.setUser(userF.get());        // 直接 get，无需超时
    vo.setInventory(invF.get());
    vo.setCoupon(couponF.get());
    return vo;
}
✅ 批量场景（100 订单）： 线程池(10 threads): 10 个线程排队，总耗时 ~2000ms
虚拟线程: 100 个虚拟线程并发，总耗时 ~200ms
10 倍提升，零代码改动。

2.4 Pinning 问题

synchronized 块内阻塞会导致虚拟线程被 pin 在载体线程上，无法卸载。

Before / After

❌ Pinning
synchronized (lock) {
    // 阻塞时虚拟线程被 pin
    // 载体线程无法释放
    httpClient.send(req, body);
}
✅ No-Pinning
lock.lock();
try {
    // 阻塞时虚拟线程可以 unmount
    // 载体线程释放
    httpClient.send(req, body);
} finally {
    lock.unlock();
}

排查方式

// 启动时加 JVM 参数
java -Djdk.tracePinnedThreads=short MyApp

// 输出示例：
// ** pinning: monitor-enter **
//     at com.example.MyService.process(MyService.java:42)

场景速查

场景是否 Pin解决方案
synchronized 块内阻塞✅ Pin改用 ReentrantLock
JNI 调用期间✅ Pin无法避免，减少 JNI
ReentrantLock 内阻塞❌ 不 Pin直接使用
⚠️ 注意事项：
• synchronized 是最常见的 pinning 源 · JNI 调用也会 pin
• 用 -Djdk.tracePinnedThreads=short 排查
• 不是所有 synchronized 都有问题（锁内没有阻塞就没影响）
• 只有锁内有 I/O/sleep/网络调用时才需要替换

2.5 什么时候不该用虚拟线程

场景建议原因
CPU 密集型计算❌ 不用不提升 CPU 计算能力
频繁 synchronized 阻塞❌ 不用Pinning 抵消优势
需要线程亲和性❌ 不用虚拟线程可能在不同载体上执行
I/O 密集型服务✅ 完美场景阻塞时自动释放载体线程
大量并发连接✅ 完美场景虚拟线程廉价，可以百万级
微服务网关/聚合层✅ 完美场景并发调用多个远程服务

2.6 配套特性

Structured Concurrency（Preview）

将并发任务视为代码块结构——任务的生命周期被限定在作用域内，异常自动传播，取消自动级联。

Before: 手动管理
Future<User> userF = ex.submit(...);
Future<Order> orderF = ex.submit(...);
try {
    User u = userF.get(5, SECONDS);
    Order o = orderF.get(5, SECONDS);
} catch (TimeoutException e) {
    userF.cancel(true);   // 手动取消
    orderF.cancel(true);  // 容易忘！
    throw e;
}
After: 结构化管理
try (var scope = new StructuredTaskScope
        .ShutdownOnFailure()) {
    var userSub = scope.fork(...);
    var orderSub = scope.fork(...);

    scope.joinUntil(deadline);
    scope.throwIfFailed();
    // 一个失败，自动取消其他

    return new Dashboard(
        userSub.get(), orderSub.get());
}

Scoped Values（Preview）

替代 ThreadLocal，不可变、自动清理、虚拟线程友好。

Before: ThreadLocal
ThreadLocal<UserId> TL = new ThreadLocal<>();
TL.set(userId);
try { doWork(); }
finally {
    TL.remove();  // 忘了就泄漏！
}
After: ScopedValue
ScopedValue<UserId> SV =
    ScopedValue.newInstance();
ScopedValue.where(SV, userId)
    .run(() -> {
        doWork();  // 可以读取 SV.get()
    });
// 作用域结束，自动清理
⚠️ 注意事项： Structured Concurrency 是 Preview（需要 --enable-preview）· ScopedValue 不可变 · ScopedValue 不能替代所有 ThreadLocal

3.1 Generational ZGC

JDK 11 ZGC JDK 21 分代 ZGC
📌 演进：JDK 11 引入 ZGC（不分代）→ JDK 21 引入分代 ZGC（JEP 439）

为什么需要分代

JDK 17 ZGC（不分代）：
  所有对象 → 同一个堆 → 全量扫描
  暂停时间 < 1ms ✅
  吞吐量一般 ❌

JDK 21 ZGC（分代）：
  短命对象 → Young Generation → 频繁但快速回收
  长命对象 → Old Generation   → 低频但深度回收
  暂停时间 < 1ms ✅
  吞吐量提升 10-30% ✅

关键数据

指标Non-Generational ZGCGenerational ZGC
GC 暂停时间< 1ms< 1ms（保持）
吞吐量基准提升 10-30%
堆内存使用较高降低 10-20%
适合场景大堆、低延迟绝大多数场景

何时需要 ZGC

你的情况建议
GC 暂停影响 SLA（P99 > 10ms）✅ 用 ZGC
吞吐量优先，暂停可接受⚠️ 用 G1 即可
堆 < 4GB⚠️ G1 通常够用
堆 > 16GB，低延迟要求✅ ZGC 最佳
# 启用方式（JDK 21 默认已是分代 ZGC）
java -XX:+UseZGC -Xmx4g -Xms4g MyApp

# 查看 GC 日志
java -XX:+UseZGC -Xlog:gc* -Xmx4g MyApp
⚠️ 注意事项： 初始内存占用较高 · 建议堆 4GB+ · 与虚拟线程配合效果最佳（虚拟线程创建大量短命对象，分代 ZGC 回收更高效）

4.2 Foreign Function & Memory API

JDK 16 Incubator JDK 21 Final
📌 演进：JDK 16 Incubator → JDK 18/19/20 Preview → JDK 21 正式定稿（JEP 442）

取代 JNI 的现代方案，纯 Java 调用 C 函数：

Linker linker = Linker.nativeLinker();
MethodHandle strlen = linker.downcallHandle(
    linker.defaultLookup().find("strlen").orElseThrow(),
    FunctionDescriptor.of(JAVA_LONG, ADDRESS)
);

try (Arena arena = Arena.ofConfined()) {
    MemorySegment str = arena.allocateUtf8String("Hello");
    long len = (long) strlen.invoke(str);  // 调用 C 的 strlen
    System.out.println(len);  // 5
}
⚠️ 注意事项： Arena 生命周期管理（关闭后访问会抛异常）· 堆外内存泄漏风险 · 大多数 Web 开发者用不到
